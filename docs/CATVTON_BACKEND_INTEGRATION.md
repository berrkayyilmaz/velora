# CatVTON Backend Integration Analysis

Version: 1.0  
Status: Planning Only  
Last Updated: 2026-07-13

## 1. Purpose

This document analyzes CatVTON's backend-relevant inference entrypoint for a
future Velora AI worker integration.

Scope inspected:

- `model/pipeline.py`
- `app.py`
- `inference.py`

This document does not implement adapters, inference, model download, checkpoint
loading in Velora, or product runtime changes.

## 3. High-Level Finding

For one backend try-on request, `app.py` is the most useful entrypoint because
it contains the single-image flow behind the Gradio demo. The Gradio UI can be
ignored, but the `submit_function` logic defines the practical request path:

1. Load person image.
2. Load garment image.
3. Prepare or generate mask.
4. Resize person, garment, and mask.
5. Create optional seeded CUDA generator.
6. Call `CatVTONPipeline`.
7. Save or return the generated PIL image.

`inference.py` is useful for understanding batch evaluation and tensor input
support, but its dataset-specific structure is not the right backend API for a
single Velora user request.

## 4. Minimal Python API Needed

A future backend worker should expose a small Python service wrapper around
CatVTON rather than invoking Gradio or dataset scripts.

Proposed minimal API shape:

```python
class CatVTONTryOnService:
    def __init__(self, config: CatVTONConfig) -> None:
        ...

    def run_try_on(self, request: CatVTONTryOnRequest) -> CatVTONTryOnResult:
        ...
```

Minimal config:

```text
base_model_path: str
resume_path_or_local_checkpoint: str
device: str
weight_dtype: torch dtype
width: int
height: int
num_inference_steps_default: int
guidance_scale_default: float
allow_tf32: bool
skip_safety_check: bool
output_dir: path
```

Minimal request:

```text
request_id: str
person_image_path: path
garment_image_path: path
garment_type: upper | lower | overall
mask_image_path: optional path
seed: optional int
num_inference_steps: optional int
guidance_scale: optional float
width: optional int
height: optional int
```

Minimal result:

```text
request_id: str
status: succeeded | failed
output_image_path: optional path
mask_image_path: optional path
duration_ms: int
error: optional string
metadata:
  model_revision
  checkpoint_path
  width
  height
  seed
  num_inference_steps
  guidance_scale
```

## 5. Required Inputs

Required for a single request:

- Person image.
- Garment image, called `condition_image` by the pipeline.
- Garment type for automatic masking:
  - `upper`
  - `lower`
  - `overall`
- Optional user-provided mask image.
- Requested output size.
- Inference steps.
- Guidance scale.
- Optional seed.

Input expectations:

- Person and garment images are loaded with PIL and converted to RGB.
- Mask is grayscale or generated by AutoMasker.
- Person and mask dimensions must match before entering the pipeline.
- Garment image is padded rather than cropped to target dimensions.
- Backend should validate file existence, image type, dimensions, and request
  ownership before invoking the model.

## 6. Preprocessing Order

The single-request preprocessing order from `app.py` should be preserved:

1. Resolve output path.
2. Create a CUDA generator when seed is provided.
3. Open person image and convert to RGB.
4. Open garment image and convert to RGB.
5. Resize and crop the person image to `(width, height)`.
6. Resize and pad the garment image to `(width, height)`.
7. If a mask is provided:
   - Load it as grayscale.
   - Binarize nonzero pixels to 255.
   - Resize and crop it to `(width, height)`.
8. If no mask is provided:
   - Run `AutoMasker(person_image, garment_type)`.
   - Read the returned `mask`.
9. Blur the mask with `VaeImageProcessor.blur(..., blur_factor=9)`.
10. Call `CatVTONPipeline`.
11. Save the result image.

Backend integration should ignore the Gradio-specific `ImageEditor` shape.
Instead, it should pass explicit file paths or resolved storage objects.

## 7. Checkpoint Loading

`app.py` initializes checkpoints in this order:

1. Resolve the CatVTON checkpoint repository with `snapshot_download`.
2. Initialize `CatVTONPipeline` with:
   - `base_ckpt="runwayml/stable-diffusion-inpainting"` by default.
   - `attn_ckpt=<downloaded CatVTON repo path>`.
   - `attn_ckpt_version="mix"` for the demo path.
   - `weight_dtype` from the mixed precision setting.
   - `device="cuda"`.
3. Initialize `AutoMasker` with:
   - `densepose_ckpt=<repo_path>/DensePose`.
   - `schp_ckpt=<repo_path>/SCHP`.
   - `device="cuda"`.

Inside `CatVTONPipeline.__init__`, the pipeline loads:

- DDIM scheduler from the base checkpoint.
- VAE from `stabilityai/sd-vae-ft-mse`.
- Optional safety checker and feature extractor from the base checkpoint.
- UNet from the base checkpoint.
- CatVTON attention adapter modules.
- Attention checkpoint weights from one of:
  - `mix-48k-1024`
  - `vitonhd-16k-512`
  - `dresscode-16k-512`

For Velora, checkpoint loading should happen once at worker startup, not once
per request.

## 8. Inference Call

The backend-relevant inference call is:

```python
result_image = pipeline(
    image=person_image,
    condition_image=cloth_image,
    mask=mask,
    num_inference_steps=num_inference_steps,
    guidance_scale=guidance_scale,
    generator=generator,
)[0]
```

`CatVTONPipeline.__call__` returns a list of PIL images. For a single request,
the backend should use the first image.

Important defaults from source:

- Demo width: `768`.
- Demo height: `1024`.
- Demo inference steps: configurable in UI, default `50`.
- Demo guidance scale: configurable in UI, default `2.5`.
- Demo seed: `42`, with `-1` meaning no seeded generator.
- Batch evaluation width: `384`.
- Batch evaluation height: `512`.

Velora should choose one worker configuration explicitly instead of mixing demo
and evaluation defaults.

## 9. Pipeline Internal Order

Inside `CatVTONPipeline.__call__`, the core order is:

1. Validate and normalize input image, condition image, and mask.
2. Convert inputs to tensors on the configured device and dtype.
3. Apply mask to the person image.
4. Encode masked person image with VAE.
5. Encode garment condition image with VAE.
6. Interpolate mask to latent dimensions.
7. Concatenate masked person and garment latents along the height axis.
8. Create random latent noise with the provided generator.
9. Set scheduler timesteps.
10. Apply classifier-free guidance when `guidance_scale > 1.0`.
11. Run the denoising loop through the UNet.
12. Split final latent back to the person side.
13. Decode latent with VAE.
14. Convert output tensor to PIL image.
15. Run safety checker unless disabled.

The worker should treat the pipeline as GPU stateful and not instantiate it per
request.

## 10. Output Image Generation

The pipeline output is a generated PIL image.

`app.py` additionally creates and saves a debug grid containing:

- Person image.
- Masked person preview.
- Garment image.
- Generated result.

For a backend API, the production-facing result should be only the generated
try-on image. Optional internal artifacts may include:

- Processed mask.
- Masked person preview.
- Debug grid.
- Benchmark metadata JSON.

Internal artifacts must remain private and follow Velora retention/deletion
rules.

## 11. `inference.py` Relevance

`inference.py` is not a good direct backend entrypoint because it is built for
dataset evaluation:

- It requires `dataset_name`.
- It expects VITON-HD or DressCode directory layouts.
- It reads pair text files.
- It builds a DataLoader.
- It writes outputs using dataset person names.

Useful backend takeaways from `inference.py`:

- The pipeline can accept tensors as well as PIL images.
- Batch processing is possible.
- `skip_safety_check=True` is used in evaluation.
- Optional repaint and concatenated output helpers exist, but they are not
  required for one Velora request.

## 12. Backend Integration Boundary

Future Velora integration should use this boundary:

```text
Fastify API
  -> Try-on job record
  -> Queue
  -> Python AI worker
  -> CatVTONTryOnService.run_try_on(...)
  -> Generated image storage
  -> Try-on result record
```

The Fastify backend should not import CatVTON directly. GPU model state belongs
in a separate Python worker process or service.

## 13. Worker Startup Plan

Worker startup should:

1. Validate license gate and config.
2. Validate CUDA availability.
3. Resolve local checkpoint paths.
4. Load `CatVTONPipeline`.
5. Load `AutoMasker`.
6. Run import and memory checks.
7. Mark worker ready only after model components are resident.

If any checkpoint is missing, the worker should fail startup rather than
downloading implicitly in production.

## 14. Request Execution Plan

For one job:

1. Load job metadata.
2. Verify consent and ownership were already enforced upstream.
3. Resolve person and garment image paths from private storage.
4. Validate image files.
5. Preprocess person and garment images.
6. Prepare or generate mask.
7. Run pipeline.
8. Save generated output.
9. Save optional internal artifacts.
10. Return result metadata.
11. Release per-request tensors and clear CUDA cache only if needed.

## 15. Risks And Open Questions

- CatVTON public materials indicate noncommercial use, so product integration is
  blocked until licensing is approved.
- The current verified Colab environment has imports verified, but app execution
  and inference are not yet verified.
- T4 memory may be tight for 768x1024 depending on precision and runtime state.
- The demo uses `attn_ckpt_version="mix"` while dataset evaluation uses
  dataset-specific versions.
- Automatic masking depends on DensePose and SCHP checkpoint availability.
- Safety checker behavior may block normal outputs, according to the demo text.
- The correct behavior for `bf16` on T4 needs empirical verification because T4
  generally favors `fp16`.
- Output retention and deletion rules must be finalized before any user-image
  experiment.

## 16. Recommended Next Step

Before implementation, run a single approved research-only smoke test outside
the app:

1. Verify model license status remains `research-only`.
2. Use synthetic or explicitly permitted images only.
3. Load the pipeline once.
4. Run one `upper` garment request with explicit seed.
5. Record runtime, VRAM, output path, errors, and output quality in
   `ml/EXPERIMENT_LOG.md`.

Do not connect this path to Velora users until license, privacy, deletion,
storage, safety, and production GPU runtime requirements are approved.
